A healthy-recipe assistant built as a **prompt agent**: a hosted model with a system prompt and a small toolset, no orchestration framework. Use this shape for quick spec-iteration loops and for systems where you don't have (or need) framework-level instrumentation.

## Source

```
examples/pipes/
├── harmful_medical_advice.md             # spec
├── health_assistant.yaml                 # hosted model only (no tools)
├── health_assistant_simulated_tools.yaml # + simulated tool responses
├── health_assistant_generated_tools.yaml # + LLM-generated tool responses
├── health_assistant_sandbox.yaml         # + Docker sandboxed tool exec
└── health_assistant_external.yaml        # + HTTP endpoint integration
```

## What you'll see

- **Spec-only iteration.** The base `health_assistant.yaml` exercises just the model + system prompt — perfect for tightening the spec before any tool work.
- **Simulated tools.** `health_assistant_simulated_tools.yaml` adds a small toolset where tool responses are canned. Lets you probe tool-routing failure modes without standing up real backends.
- **Generated tools.** `health_assistant_generated_tools.yaml` has the tool responses themselves generated by an LLM at runtime — useful for exploring failure modes that depend on adversarial tool output (e.g. prompt-injection-in-tool-result).
- **Sandbox path.** `health_assistant_sandbox.yaml` executes tool code in a Docker sandbox; `health_assistant_external.yaml` hits a real HTTP endpoint.

## Run it

```bash
python -m pip install -e ".[otel]"
p2m run --config examples/pipes/health_assistant_simulated_tools.yaml
```

## When to pick this shape

- You don't have a framework agent yet — you have a system prompt and a model.
- You're iterating on the spec or the prompt itself and want a tight loop.
- You want to probe tool-routing or adversarial-tool-output failure modes without building real tool backends.

Once you've moved to a real framework agent, switch to the [Travel Planner OTel pattern](/adaptive-eval/examples/travel-planner-otel/) — same eval surface, dramatically more observability.

## Notes on adversarial probing

Hosted models like `azure/gpt-5.4*` are aligned. Adversarial probes against them produce conservative results: low `policy_violation_rate`, high `overrefusal_rate`. That's not necessarily "the agent is safe" — it can be the alignment layers absorbing the probe. See [Risks & limitations](/adaptive-eval/reference/risks/) for full interpretation guidance.
