# Reading Results

Every run writes local artifacts under `artifacts/results/<suite>/<run>/`.

## Directory layout

```text
artifacts/results/<suite>/
├── suite.json
├── taxonomy.json
├── test_set.jsonl
└── <run>/
    ├── manifest.json
    ├── config.yaml
    ├── inference_set.jsonl
    ├── scores.jsonl
    └── metrics.json
```

## What to read first

1. `metrics.json` - aggregate pass/fail rates by judge dimension and behavior category.
2. `scores.jsonl` - individual verdicts with justification and evidence.
3. `inference_set.jsonl` - inference outputs (conversations or agent actions), plus trace references when configured.
4. `config.yaml` - the exact config snapshot for reproducibility.

## How to debug a failure

Start with a failing row in `scores.jsonl`.

Look for:

- which judge dimension failed
- which behavior category was targeted
- the evidence turns cited by the judge
- any tool calls or trace references

Then open the matching transcript and inspect:

- user turns generated by the tester
- target responses
- tool calls and tool results
- trace references from the OTel backend

## What makes a useful verdict

A useful verdict should:

- cite specific conversation turns
- explain why the behavior did or did not match the spec
- distinguish final-answer failures from trace/tool failures
- avoid relying on generic benchmark criteria that were not in your spec

If verdicts are too vague, tighten the judge dimension description and rubric.
